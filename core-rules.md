# core-rules.md — 核心执行规则

> 所有场景通用，每次会话必读。

---

## 优先级（冲突时）

```
正确性 > 可维护性 > 性能 > 简洁性 > 速度
```

目标模糊时，先做"正确的最小版本"，不做"完整但可能偏的版本"。

---

## 执行阈值

**功能开发模式**（改动在当前模块内）：
- 直接执行，完成后说明假设
- 影响范围超出当前模块时升级判断

**模块负责人模式**（涉及对外接口/本模块规范）：
- 有两种合理方案 → 列出 tradeoff，让用户选
- 影响超过 3 个文件 → 先列改动计划（机械性改动例外）
- 跨模块边界时：① 写验收标准 → ② 列改动计划 → ③ 执行

**架构把控模式**（跨模块、依赖、配置、全局结构）：
- 任何改动前先说明影响范围
- 不可逆操作必须确认后执行

**真正需要暂停的情况**：
- 需求有根本性歧义（A 和 B 两个方向都说得通）
- 改动不可逆且影响范围不清楚
- 指令和当前代码状态明显冲突

> 能写出合理假设就执行，写不出来就问。不用"可能有风险"暂停，用"执行方向有根本歧义"。

---

## 改动范围原则

- 只动需要动的，不顺手"优化"周边代码
- 不要在一次改动里修多个不相关的问题
- 影响范围超出预期时：执行前说明，不是执行后汇报

---

## 文件操作原则

- 单文件改动：直接执行
- 多文件批量改动：先列改动计划，确认后执行
  - 例外：机械性改动（统一重命名、批量格式化、联动修改同一接口调用）→ 直接执行并说明
- 删除操作：临时文件/日志直接删；源码文件需确认

---

## Kotlin/Android 架构硬约束

没有就会出错，每次写代码必须遵守：

- UseCase 必须定义接口，实现类加 `Impl` 后缀
- ViewModel 不持有 View 引用；UI 状态通过 `StateFlow`/`LiveData` 单向流动
- Repository 层通过接口隔离，不直接依赖具体数据源实现
- 禁止 `!!` 非空断言；用 `?: return`、`let`、`requireNotNull` 替代
- 协程 Scope 只在 `viewModelScope` 或明确的生命周期组件中管理
- 不随意引入新依赖；需要时说明理由

---

## Bug 分析格式

每次分析 bug 必须包含以下四项：

```
🔍 根因：[一句话定位，精确到代码层/调用链]
📎 证据：[文件名#行号 / 日志片段 / 调用栈]
🔧 修复：[具体改法，含改哪里、改成什么]
⚠️  关联风险：[本次修复可能影响的其他地方，没有则省略]
```

根因不确定时标注 `[待验证]`，并说明验证方法；禁止写"可能是 X"而不给证据。
